# Decision: Tool awareness layer + model routing for every agent

**Date:** 2026-05-07
**Status:** Accepted
**Affected components:** `src/agent_os/orchestrator/{catalog,tier_classifier,tool_planner,model_planner,plan_card,adapters/plan_overrides}.py`, `vault/skills/active/tools/`, `src/agent_os/orchestrator/config/{models,tiers}.yaml`, identity packs, `cli.py`

## Context

The super-agent stack ships ~15 specialist tools (OpenClaw, OpenSwarm, browser-use, Agent Zero, Computer Use, Claude subagents, Codex CLI, Aider, Claude Managed, E2B, Exa, LiveKit, Terminal, Composio, hermes-self) plus a model fleet (Kimi K2, Claude 4.7, Sonnet 4.6, GPT-5.5, DeepSeek v4 Pro, Gemini 2.5). Two text docs already capture the routing intent:

- `docs/routing-intelligence-contract.md` — when to use each tool/model/backend
- `docs/builder-tool-architecture.md` — the layered "tool roles" map

Both are authoritative and good. **But neither is read by anything programmatic.** Hermes' router has no per-tool descriptions to score against. `vault/skills/active/` was empty in main. `agent-os route` only returns a runtime name with no rationale, no cost estimate, no alternatives, no confirmation gate.

The user's specific ask: every agent should be **fully aware of every tool**, surface its **tool plan at task entry**, and balance **autonomous action vs explicit confirm** by task risk.

## Decision

Build a five-layer tool-awareness system:

1. **Per-tool reusable SKILL.md files** in `vault/skills/active/tools/` — one per tool, structured frontmatter (tier/cost_class/risk_class/preferred_models/category/description) plus a body with when-to-use, when-not-to-use, alternatives, examples. Hermes' router scores each on every prompt.

2. **Machine-readable tool catalog** at `vault/graph/tool-catalog.yaml` — auto-generated by `agent-os catalog` from the per-tool SKILL.md files + runtime manifests + identity packs + model registry. Read by the planner, the dashboard, `/explain`, and any future Hermes integration.

3. **Tier classifier** (`tier_classifier.py`) — heuristic that classifies each job into Tier 1 (autonomous), Tier 2 (plan card + 3s grace), or Tier 3 (hard stop). Rules live in `config/tiers.yaml` and are tunable without code changes.

4. **Tool planner** (`tool_planner.py`) — scores the catalog against a job, filtered by the agent's identity bundle, returns a `ToolPlan` with primary tool + alternatives + tier + cost/time estimates + model recommendation.

5. **Plan card emitter + override surface** (`plan_card.py` + `adapters/plan_overrides.py`) — formats the plan per tier (1-line banner / 4-line card / hard-stop), parses `/cancel`, `/use <tool>`, `/why`, `/plan on|off`, `/tier N`, `YES`.

Plus the **model layer**: `config/models.yaml` registers Kimi K2, Claude 4.7, Sonnet 4.6/4.7, GPT-5.5, **DeepSeek v4 Pro** (not v3), Gemini 2.5. `model_planner.py` picks per task class with dual-frontier review (gpt-5.5 ↔ claude-opus-4.7) for architecture / debug / security / auth / tests / deploy work.

## Why tier-gated transparency (not always-emit, not on-demand)

- **Always-emit** would be chatty for the ~80% of tasks that are tier 1. Visual noise, latency cost.
- **On-demand** (`/plan` to see the plan) puts the burden on the user to remember to ask. Defeats "build familiarity through repetition".
- **Tier-gated** matches the user's instinct: read-ony quick stuff just runs (with a banner), substantive work pauses for a plan + 3s grace, destructive/expensive work hard-stops. The user can flip to always-emit per-session with `/plan on`.

## Why per-tool SKILL.md (not grouped by category)

- The router scores skills by description match. Per-tool descriptions give it the granularity to distinguish openclaw vs openswarm vs browser_use without false-positive matches.
- The maintenance cost (~15 small files) is low and grows linearly only when new tools are added — exactly when the catalog should grow.
- Per-tool files also serve as the **human-readable "what does this tool do" reference** — the user reads the same files the router does. No drift between docs and code.

## Why a runtime catalog (not just docs)

- The planner needs a single, structured object to score against. Reading 15 markdown files at every plan() call is wasteful.
- The dashboard's OpenSwarm tile precedent (`vault/graph/openswarm.json`) showed this pattern works — a vault-resident JSON/YAML snapshot the dashboard reads.
- Catalog regeneration is cheap (~50ms) and triggered by `agent-os catalog`, by the upgrader nightly, or on any skill edit (Phase F.6 file-watch).

## Why the model layer ships in the same plan

- Tool selection and model selection answer the same parent question: "what's the right execution shape for this task?"
- The plan card has space for both (`Tools: X` + `Model: Y`), making the trade-off visible.
- Adding the model layer separately would mean two planner overhauls in two PRs. Doing them together is cleaner and matches `routing-intelligence-contract.md`'s existing structure (it already has a model section).

## Identity-pack tool ceilings

Each identity now declares `tools_allowed`, `tools_denied`, `default_tier_ceiling`. The COO can use OpenSwarm for client decks but not Terminal for destructive infra. A COO asking for a tier-3 deploy gets a `🚫 Blocked: ... requires primary_hermes approval` instead of an outright refusal — escalation path is explicit.

## Consequences

- Every Hermes session at boot loads the catalog summary into context. Sub-agents inherit. Identity packs filter.
- `agent-os plan` becomes the verification command for any non-trivial task (precedent: `agent-os route` already exists).
- Cost data lands in run artifacts via the new `RuntimeResult.plan` field — future analytics can answer "which tools cost what" per identity.
- The 14 old framework wrapper repos remain frozen (per ETHOS.md). All future tool additions are SKILL.md + manifest entries here, not new wrappers.
- Models like DeepSeek v4 Pro (cheap mechanical work) and Kimi K2 (cheap long-context coding) get explicit homes in the registry — they were missing from the prior text-only docs.

## Verification

- 5 CLI commands working: `agent-os plan`, `agent-os tier`, `agent-os tool <name>`, `agent-os catalog`, `agent-os models`
- 15 per-tool SKILL.md files indexed by catalog → 15 tools in `vault/graph/tool-catalog.yaml`
- 7 models registered in `config/models.yaml` (DeepSeek v4 Pro confirmed)
- 3 identity packs (coo/gtm/head_of_ops) with `tools_allowed/denied/ceiling`
- 50+ new unit tests across catalog / tier / planner / model / card / overrides; existing 128 still pass
- End-to-end smoke: T1 banner + T2 card + T3 hard-stop + COO ceiling block all render correctly via `agent-os plan`

## Out of scope (explicit follow-ups)

- **Embedding-based scoring** — current scoring is tag-match + keyword bonus. Phase F.6 adds vector similarity for prompts that don't tag-match cleanly.
- **Hermes turn-loop integration** — the planner is callable today via `agent-os plan` and via the `plan_for()` helper in job_router. When Hermes' turn loop is properly wired (per `docs/EXECUTION-PLAN.md` Stage 2), the `before_turn` hook is one line. Phase F prepared the seam; doesn't ship that wiring.
- **Channel-specific rendering** — current plan_card renders Markdown. Slack-block-kit, Telegram-formatted, voice-friendly variants are Phase F.7.
- **MCP registry skill** — `mcp_registry` runtime exists in the upgrader streams; per-tool SKILL.md for it lands when MCP servers are dynamically discovered (not just registered).
- **Cost analytics rollup** — the `RuntimeResult.plan` field is now persisted; the dashboard tile that reads it across the fleet is Phase F.8.
