# Tool Awareness Layer + Model Routing — Full Handoff

_Last updated: 2026-05-07_
_Scope: incorporate this layer into another project (Hermes-style or otherwise)._

---

## 0. TL;DR

This is a self-contained layer that gives any agent stack:

1. **Awareness of every tool/runtime it has** — when to use each, when not to, what it costs, what it risks, and which model should drive it.
2. **A tool plan emitted at task entry** — "I'll use X with model Y because Z" surfaced *before* anything runs.
3. **Tier-gated UX** — Tier 1 banner / Tier 2 plan card + 3s grace / Tier 3 hard stop with explicit `YES`.
4. **Identity ceilings** — different roles get different tool bundles + tier ceilings; escalation paths are explicit, not refusals.
5. **Model routing** — seven models registered (Claude Opus 4.7, Sonnet 4.7/4.6, GPT-5.5, Kimi K2, DeepSeek v4 Pro, Gemini 2.5 Pro) with task-class tagging and dual-frontier review pairs.
6. **Override surface** — `/cancel /use /why /plan on|off /tier N YES` work in any channel.

It's ~1100 lines of Python + 15 markdown skills + 3 YAML configs. No DB. No network calls in the planner. Catalog is a YAML snapshot regenerated by one CLI command.

---

## 1. What this is, conceptually

**Problem.** Most agent stacks end up with one of two failure modes:
- **Always autonomous** — the agent silently picks tools and runs. The user has no idea what just happened, can't intervene before money is spent, and can't audit afterward.
- **Always confirm** — every step requires approval. The user is buried in confirmations for trivial reads. Eventually they rubber-stamp everything and the safety theater stops working.

**Solution.** Classify each task into a tier and behave appropriately:

| Tier | Trigger | Behavior |
|---|---|---|
| **1** | Read-only, idempotent, cheap (search, status, explain, lookup, manifest) | Banner: `⚡ using openswarm · claude-opus-4.7`. Run. |
| **2** | Mutates state, substantive (build, write, generate, send) | 4-line plan card + 3-second grace window for `/cancel` or `/use <tool>`. Then run. |
| **3** | Destructive, public, expensive (deploy, delete, force-push, send-email, >\$1.00) | Hard stop. Show plan. Require uppercase `YES` to proceed. |

The tiers are tunable in `tiers.yaml` — no code changes needed to adjust which tags lift to Tier 3.

**The plan card** carries five things in ~4 lines: task summary, primary tool + alternatives, model recommendation, tier + cost + time + grace, override commands. The user can:

- Watch it scroll past at Tier 1 (banner only)
- React during the 3s window at Tier 2 (`/cancel`, `/use openclaw`, `/why`)
- Confirm or refuse at Tier 3 (uppercase `YES` or `/cancel`)

**Per-tool SKILL.md** files are how the planner knows what each tool does. They live in `vault/skills/active/tools/<name>.md` and have a structured frontmatter (tier, cost_class, risk_class, preferred_models) plus a body that explains *when to use* and *when NOT to* in human terms. The router reads the description for semantic match; the planner reads the frontmatter for scoring; you read the body when you want to know what a tool actually does.

**Identity packs** are the access-control layer. Each identity (e.g. `coo`, `gtm`, `head_of_ops`) has `tools_allowed`, `tools_denied`, and `default_tier_ceiling`. The planner filters the catalog before scoring — a COO can't accidentally trigger a Terminal-rooted destructive command, and a deploy attempt by the COO surfaces as `🚫 Blocked: requires primary_hermes approval` instead of an outright refusal.

## 3. Architecture — how the pieces fit

```
┌───────────────────────────────────────────────────────────────────┐
│ User message arrives (Slack / Telegram / CLI / web / voice)        │
└───────────────────────────────────────────────────────────────────┘
                                │
                                ▼
                     ┌─────────────────────┐
                     │  Job(prompt, tags)  │  ← dataclass already in job_router.py
                     └─────────────────────┘
                                │
                                ▼
┌───────────────────────────────────────────────────────────────────┐
│ tool_planner.plan(job, identity="coo")                             │
│                                                                    │
│  1. catalog.build_catalog()                                        │
│     ├── load_skills()           (15 SKILL.md frontmatter)         │
│     ├── load_runtime_manifests()(per-runtime manifest.yaml)       │
│     ├── load_identities()       (3 identity packs)                │
│     └── load_models()           (7-model registry)                │
│  2. _filter_by_bundle()         (apply identity's tools_allowed)  │
│  3. for each tool: _score_tool()                                   │
│       score = ROUTER_MATCH (50)                                    │
│             + TAG_HIT (10) per matching tag                        │
│             + KEYWORD (5) per matching keyword                     │
│             + TIER_PENALTY (-2 / -8 for tier 2/3)                  │
│  4. tier_classifier.classify(job, cost, minutes)                   │
│       returns TierDecision(tier=1|2|3, reason, signals)            │
│  5. model_planner.pick_model(task_class, preferred=[...])          │
│       returns (model_id, reason)                                   │
│  6. Apply identity ceiling — if tier > ceiling → blocked_reason   │
│  7. Return ToolPlan(...)                                           │
└───────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌───────────────────────────────────────────────────────────────────┐
│ plan_card.render(plan, channel="markdown")                         │
│  ├── tier 1 → 1-line banner                                        │
│  ├── tier 2 → 4-line card                                          │
│  ├── tier 3 → hard-stop card with "reply YES"                      │
│  └── blocked → "🚫 Blocked: requires primary_hermes approval"     │
└───────────────────────────────────────────────────────────────────┘
                                │
                                ▼
              ┌────────────────┴────────────────┐
              ▼                                  ▼
     ┌─────────────────┐                 ┌──────────────────┐
     │ User reaction    │ ◄─ override ─► │ plan_overrides   │
     │ during grace     │                │ .parse(text)     │
     │ (Tier 2: 3s)     │                │ → Override       │
     │ or wait (Tier 3) │                │   kind=cancel|   │
     └─────────────────┘                 │        use|tier| │
              │                          │        confirm   │
              ▼                          └──────────────────┘
     ┌─────────────────┐
     │ runtime invoke()│ ← existing flow, untouched
     │ writes Runtime  │   (just adds `plan` field to result artifact)
     │ Result + plan   │
     └─────────────────┘
```

**Key invariants:**
- The planner is **pure** — no I/O, no LLM calls, no network. Catalog is loaded once.
- The catalog is **a snapshot** at `vault/graph/tool-catalog.yaml`. Regenerate with `agent-os catalog` (~50ms). The planner reads the snapshot, not the source files.
- The model planner is **rule-based + deterministic**. No LLM call to pick a model.
- The override surface is **case-sensitive on `YES`** (uppercase only) — prevents conversational `yes` replies from auto-confirming Tier 3.

## 4. Integration into another project

Six paths, in order of how invasive they are. Pick the one that matches your stack.

### Path 1 — Standalone Python library (simplest)

If you just want the planner as a library someone calls from anywhere:

1. Copy `src/agent_os/orchestrator/{catalog,tier_classifier,tool_planner,model_planner,plan_card}.py` and `adapters/plan_overrides.py` into your project. Adjust the package path in imports if needed.
2. Copy `src/agent_os/orchestrator/config/{tiers,models}.yaml` and `config/identities/*.yaml`.
3. Copy `vault/skills/active/tools/*.md`.
4. Provide a `Job` dataclass with `prompt: str` + `tags: set[str]` + `estimated_minutes: int | None`. We use the one in `adapters/job_router.py`; copy it or define your own with the same shape.
5. Call:

   ```python
   from agent_os.orchestrator import tool_planner, plan_card
   from agent_os.orchestrator.adapters.job_router import Job

   job = Job(prompt="make me a deck", tags={"multi-deliverable"})
   plan = tool_planner.plan(job, identity="primary_hermes")
   print(plan_card.render(plan))
   ```

That's it. No turn loop, no channel adapter — you call this from wherever your code already decides what to run.

### Path 2 — CLI integration

If your project has an existing CLI:

1. Path 1 above.
2. Lift the five subcommands out of `src/agent_os/cli.py` (`plan`, `tier`, `catalog`, `tool`, `models`) into your CLI's `argparse` setup.
3. Done. Users get `your-cli plan --tags X --prompt "..."`.

### Path 3 — Turn-loop hook (Hermes-style agents)

If your agent has a "process incoming message → decide what to do → run" loop:

1. Path 1 above.
2. Add a `before_turn(job)` hook that calls `tool_planner.plan(job)` and emits the plan card to the channel.
3. For Tier 2, start a 3-second timer. If a `/cancel` or `/use` arrives, parse it with `plan_overrides.parse(text)` and act. Otherwise execute.
4. For Tier 3, block until `YES` arrives. If `/cancel`, abort. Anything else → re-emit the plan and wait again.
5. For Tier 1, emit the banner and execute immediately.

The planner is one line — `plan = tool_planner.plan(job, identity=current_identity)`. The hook is purely UX; the actual tool/runtime invocation stays in your existing dispatcher.

### Path 4 — Runtime adapter integration

If you also want the plan persisted with the run artifact:

1. Path 3 above.
2. In your `RuntimeResult` dataclass (or equivalent), add `plan: dict | None = None`.
3. In your runtime adapters' `invoke(job, *, plan=None)`, accept the optional plan and stash it in the result. The vault writer dumps it alongside the result artifact.

This means every run logs `why` it was chosen (primary tool, alternatives, scoring, tier reason) — your dashboard tile can answer "which tools did the COO use last week?" directly from the artifacts.

### Path 5 — Slack/Telegram/web channel adapter

If you want the plan card and overrides available across channels:

1. Path 4.
2. Each channel adapter:
   - On incoming message, build a `Job` from the message + identity inferred from the channel.
   - Call `tool_planner.plan(job)`.
   - Render via `plan_card.render(plan, channel="markdown")` — the markdown output works as-is in Slack and Telegram with no transformation.
   - For Tier 2, schedule a 3s grace task, then execute unless cancelled.
   - For Tier 3, post the card and wait for `YES` or `/cancel` in the same thread.
3. Override commands (`/cancel`, `/use`, `/why`, `/plan`, `/tier`, `YES`) are parsed by `plan_overrides.parse()` — same parser everywhere.

### Path 6 — Full incorporation (this repo's default)

If you have a Python orchestrator that already manages identities, runtimes, and a vault:

1. Copy everything in §2 verbatim.
2. Wire the `plan_for()` helper in your job router (one-line lazy import).
3. Add the new CLI subcommands.
4. Add the optional `plan` field to your `RuntimeResult`.
5. Update your README with the §3 of this document and link to the decision record.

This is what landed in PR #2 here.

## 5. Configuration knobs (tunable without code changes)

### `tiers.yaml`

```yaml
tier_3_tags:                      # any of these in job.tags → tier 3
  - deploy
  - delete
  - destroy
  - drop-table
  - force-push
  - publish
  - send-email
  - rm-rf

tier_3_pairs:                     # both tags present → tier 3
  - [production, write]
  - [production, deploy]
  - [main, force-push]

tier_1_tags:                      # any of these → tier 1 (unless mutation tag also present)
  - read
  - search
  - explain
  - status
  - lookup

tier_3_cost_threshold_usd: 1.00   # estimated cost above this → tier 3
tier_2_cost_threshold_usd: 0.05
tier_1_cost_threshold_usd: 0.05

tier_3_estimated_minutes: 60      # estimated time above this → tier 3
tier_2_estimated_minutes: 5

mutation_tags:                    # tags that lift a job out of tier 1
  - write
  - edit
  - build
  - generate
  - send
  - create
  - update
  - commit
  - push
```

**To make a new tag destructive:** add it to `tier_3_tags`. No code change.
**To make Tier 1 stricter on cost:** lower `tier_1_cost_threshold_usd`.
**To prevent a tag combo:** add it to `tier_3_pairs`.

### `models.yaml`

```yaml
models:
  claude-opus-4.7:
    backend: anthropic
    cost_per_mtok_in: 15.00
    cost_per_mtok_out: 75.00
    context_window: 200000
    task_classes:
      - architecture
      - content
      - design
      - brand-voice
      - review

  kimi-k2:
    backend: openrouter
    cost_per_mtok_in: 0.15
    cost_per_mtok_out: 2.50
    task_classes:
      - coding_background
      - mechanical_coding
      - long_context_coding
  # ... 5 more

dual_frontier_pairs:               # draft → review pair
  - [claude-opus-4.7, gpt-5.5]
  - [gpt-5.5, claude-opus-4.7]

dual_frontier_task_classes:        # tasks that always trigger draft+review
  - architecture
  - debugging
  - security
  - authentication
  - deployment
  - high_risk_coding

```

**To register a new model:** add a key under `models:` with `cost_per_mtok_in/out` + `task_classes`. The catalog picks it up on the next `agent-os catalog`.
**To change the default:** edit `default_model`.
**To add a new dual-frontier task class:** append to `dual_frontier_task_classes`.

### Identity packs (`config/identities/<name>.yaml`)

```yaml
name: Alex
title: Chief Operating Officer
default_model: ${HERMES_DEFAULT_MODEL}
slack_channels: [ops, exec]
tools_allowed:
  - hermes_self
  - exa
  - composio
  - openswarm
  - livekit
  - claude_subagents
tools_denied:
  - terminal
  - claude_managed
  - e2b
default_tier_ceiling: 2
```

- `tools_allowed` — explicit allowlist; if empty, infers "all minus denied"
- `tools_denied` — always wins over allowed (intersection is denied)
- `default_tier_ceiling` — max tier this identity can reach autonomously (Tier 3 above the ceiling produces a `🚫 Blocked` plan with the escalation hint)

`primary_hermes` has implicit access to everything (bypasses the bundle filter and ceiling).

## 6. Extending the system

### Add a new tool

1. Create `vault/skills/active/tools/<name>.md` with frontmatter:
   ```yaml
   ---
   name: <name>
   runtime: <runtime_name>
   tier: 2                          # default tier when invoked
   category: <category>
   cost_class: low|medium|high
   risk_class: low|medium|high
   preferred_models: [<model_id>, ...]
   mcp_or_native: native|mcp
   description: One-paragraph "use when..." — the router scores against this.
   ---
   ```
2. Add a body with the four standard sections: `When to use`, `When NOT to use`, `Alternatives`, `Cost & latency`, `Examples`, `See also`.
3. (Optional) Add the tool to the `route()` function in `adapters/job_router.py` if it has tag-driven shortcuts.
4. (Optional) Add it to identity packs that should have access.
5. Run `agent-os catalog` to regenerate `tool-catalog.yaml` + cheatsheets.

The planner picks it up automatically.

### Add a new model

1. Add an entry to `config/models.yaml` (see §5).
2. (Optional) Add it to per-tool `preferred_models` lists.
3. Run `agent-os catalog`.

The planner's `pick_model()` will route to it on matching `task_classes`. No code change.

### Add a new identity

1. Create `config/identities/<name>.yaml` (see §5).
2. Run `agent-os catalog`.

The planner accepts `identity="<name>"` immediately.

### Tighten or loosen a tier rule

Edit `config/tiers.yaml`. No code change. The classifier is loaded fresh each invocation.

## 8. Known limitations + future work

These are explicitly out of scope for the layer as shipped. Document them in your project so you know what NOT to expect:

- **Embedding-based scoring.** Scoring today is tag-match + keyword bonus. For prompts that don't tag-match cleanly, the router falls back to `hermes_self`. Phase F.6 adds vector similarity over the SKILL.md descriptions for higher-fidelity routing on free-text prompts. Trivial to add — load embeddings at catalog build time, replace `_score_tool` with cosine + tag bonus.
- **Cost class is static.** Some tools (openswarm, claude_managed) have wide cost ranges depending on what's done. The catalog ships a single `cost_class` per tool. Phase F.6 (later) reads recent run artifacts to estimate per-tool cost dynamically — `vault/runs/<runtime>/*.yaml` already carries `cost_usd`.
- **Channel-specific rendering.** `plan_card.render()` produces markdown. Slack-block-kit, Telegram-formatted, voice-friendly variants are Phase F.7. The markdown output renders fine in both Slack and Telegram as-is; it's the rich-block stuff that's not implemented.
- **MCP registry skill.** A per-tool SKILL.md for `mcp_registry` exists in spec but isn't shipped — it lands when MCP servers are dynamically discovered (not just registered).
- **Cost analytics rollup.** `RuntimeResult.plan` field is persisted, but the dashboard tile that aggregates "which tools cost what per identity per week" is Phase F.8. The data is there; the visualization isn't.
- **Hermes turn-loop integration.** The planner is callable today via `agent-os plan` and the `plan_for()` helper. When the project's turn loop is wired (Stage 2 of `docs/EXECUTION-PLAN.md`), the `before_turn` hook is one line. The seam is prepared; the wiring isn't shipped.

## 9. Why each design choice (the rationale you can defend in review)

**Why tier-gated transparency, not always-emit or on-demand?**
- *Always-emit* would be chatty for the ~80% of tasks that are Tier 1 — visual noise, latency cost.
- *On-demand* (`/plan` to see the plan) puts the burden on the user to remember. Defeats "build familiarity through repetition".
- *Tier-gated* matches instinct: read-only stuff just runs (with a banner), substantive work pauses for a plan + grace, destructive/expensive work hard-stops. The user can flip to always-emit per-session with `/plan on`.

**Why per-tool SKILL.md, not grouped by category?**
- The router scores skills by description match. Per-tool descriptions give it the granularity to distinguish openclaw vs openswarm vs browser_use without false-positive matches.
- Maintenance cost (~15 small files) is low and grows linearly only when new tools are added — exactly when the catalog should grow.
- Per-tool files also serve as the **human-readable "what does this tool do" reference** — the user reads the same files the router does. No drift between docs and code.

**Why a runtime catalog (YAML snapshot), not just docs?**
- The planner needs a single, structured object to score against. Reading 15 markdown files at every `plan()` call is wasteful.
- The dashboard tile precedent (`vault/graph/openswarm.json`) showed this pattern works — a vault-resident JSON/YAML snapshot the dashboard reads.
- Catalog regeneration is cheap (~50ms) and triggered by `agent-os catalog`, by the upgrader nightly, or on any skill edit (file-watch is Phase F.6).

**Why does the model layer ship in the same plan?**
- Tool selection and model selection answer the same parent question: "what's the right execution shape for this task?"
- The plan card has space for both (`Tools: X` + `Model: Y`), making the trade-off visible.
- Adding the model layer separately would mean two planner overhauls in two PRs.

**Why are identity ceilings explicit blocks instead of refusals?**
- Refusals are dead-ends — the user gets stuck without knowing the path forward.
- Explicit blocks (`🚫 Blocked: identity 'coo' has default_tier_ceiling=2; this task is tier 3. Requires primary_hermes approval.`) tell the user exactly what to do: switch identity or escalate. The escalation path is part of the UX, not a hidden policy.

**Why is `YES` case-sensitive uppercase only?**
- A casual `yes` in a follow-up reply shouldn't auto-confirm a Tier 3 hard stop. Uppercase `YES` is a deliberate keystroke — it can't be triggered by reflex.

**Why is the planner pure (no I/O, no LLM)?**
- Determinism. The same job + catalog + identity always produces the same plan. Tests are easy. Production behavior is predictable.
- Speed. A plan call is ~5ms. You can call it on every incoming message without thinking about cost.

## 10. Quick-reference cheatsheet

```python
# Build a plan from a Job
from agent_os.orchestrator import tool_planner, plan_card
from agent_os.orchestrator.adapters.job_router import Job

job = Job(prompt="research AI agents and turn it into a deck",
          tags={"multi-deliverable", "investor-pitch"},
          estimated_minutes=15)
plan = tool_planner.plan(job, identity="primary_hermes")
print(plan_card.render(plan))                 # markdown
print(plan_card.render(plan, channel="json"))  # JSON for API consumers
print(plan_card.render(plan, channel="why"))   # /why explanation

# Parse a user's override
from agent_os.orchestrator.adapters import plan_overrides
override = plan_overrides.parse("/use openclaw kimi-k2")
# Override(kind='use', raw='/use openclaw kimi-k2', tool='openclaw', model='kimi-k2', ...)

# Classify a tier without invoking
from agent_os.orchestrator import tier_classifier
decision = tier_classifier.classify(job, cost_usd=0.40, estimated_minutes=10)
# TierDecision(tier=2, reason='mutates state OR substantive work', signals=[...])

# Pick a model directly
from agent_os.orchestrator import model_planner
model_id, reason = model_planner.pick_model(
    task_class="architecture",
    preferred=["claude-opus-4.7", "gpt-5.5"],
```
